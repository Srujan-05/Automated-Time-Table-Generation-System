================================================================================
                    TIMETABLE GENERATION SYSTEM - CLASS DOCUMENTATION
================================================================================

This document describes all classes and data structures used in the Automated 
Time Table Generator system. The system uses a Genetic Algorithm to optimize 
university course scheduling across multiple constraints and objectives.

================================================================================
                              MAIN CLASSES
================================================================================

1. GASearch (Genetic Algorithm Search)
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Main genetic algorithm engine for timetable generation
   
   Attributes:
   - time_slots (dict or int): Time slots per day. Can be:
                              * Integer: same slots for all days (auto-converts to dict)
                              * Dict: day-specific slots {Monday: 9, Tuesday: 9, ..., Wednesday: 5}
   - courses (list): List of all CourseInstance objects to schedule
   - preference_bins (dict): Maps time slots to preference bins {slot: bin_id}
   - objective_function_weights (list): Weights for three penalty terms [w1, w2, w3]
   - rooms (list): List of available Room objects
   - population_size (int): Size of GA population (default: 20)
   - generations (int): Number of generations to run (default: 100)
   - mutation_rate (float): Probability of mutation (default: 0.1)
   
   Methods:
   - create_population(): Generates initial valid population of timetables using smart initialization
   - fitness(time_table): Evaluates fitness using weighted objective functions
   - select_parents(): Selects parents for crossover based on fitness
   - crossover(time_table1, time_table2): Creates offspring from two parents
   - mutate(time_table): Applies random mutations to a timetable
   - check_constraints(time_table): Validates all 8 hard constraints
   - run(): Main GA loop to evolve solutions
   
   Private Methods:
   - _ptp_objective_function(): Course Time Preference penalty using CourseInstance.preference_bin
   - _rtr_objective_function(): Room-to-room travel distance penalty calculation
   - _course_stability_objective_function(): Course session consistency penalty
   - _generate_smart_timetable(): Creates initial timetable using smart greedy algorithm
   - _can_add_course(): Checks if two courses can be scheduled in same time slot

================================================================================

2. CourseInstance (Course Instance)
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Represents a single course instance (specific course session for specific student group)
   
   Attributes:
   - id (int): Unique identifier for this course instance
   - course_id (str): Unique identifier for the parent course (e.g., "MA101", "CS201")
   - session_type (str): Type of session - "lecture", "tutorial", or "lab"
   - instructor (Instructor): The professor teaching this course instance
   - room (Room): The room assigned for this course (may be reassigned during scheduling)
   - student_grp (StudentGroup): The student group attending this course instance
   - slots_req (int): Number of time slots required for this session
   - slots_continuous (boolean): Whether slots must be consecutive (e.g., True for 2-hour labs)
   - preference_bin (int): THIS COURSE'S preferred time bin (1=morning, 2=noon, 3=evening)
                          ⚠️  INDEPENDENT PER INSTANCE - NOT inherited from instructor!
   - lecture_consecutive (boolean): If True, allows multiple lectures on same day; if False (default), max 1 lecture/day
   - course_credits (float): Credit hours for this instance (auto-calculated)
   
   Key Design:
   - Each course instance has its OWN preference_bin, independent of the instructor's preference
   - Example: Prof Kumar teaches CS201-Lecture (preference_bin=1) and MA101-Tutorial (preference_bin=3)
             These are two separate instances with different time preferences
   - The instructor's personal preference is stored in the Instructor class separately
   
   Note: Each course instance is unique. If MA101 is taught by Prof A to CSE-2027
         and Prof B to ME-2027, these are two separate instances with potentially
         different preference_bins for each.

================================================================================

3. Instructor (Professor)
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Represents a professor/instructor
   
   Attributes:
   - id (int): Unique identifier for the instructor
   - name (str): Name of the instructor
   
   Important:
   - Instructors are simple identifiers with no time preferences
   - Each CourseInstance they teach has its OWN preference_bin
   - Example: Prof Kumar teaches:
             * CS201-Lecture instance with preference_bin=1 (morning)
             * MA101-Tutorial instance with preference_bin=3 (evening)
   
   Constraints:
   - Cannot teach two different courses in the same time slot
   - Available at all times (no blocked slots)

================================================================================

4. Room
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Represents a physical classroom/lab space
   
   Attributes:
   - id (str/int): Unique identifier for the room
   - x (float): X coordinate for room location (for distance calculation)
   - y (float): Y coordinate for room location (for distance calculation)
   - z (float): Z coordinate for room location (for distance calculation)
   - capacity (int): Maximum number of students the room can accommodate
   - is_lab (boolean): True if room is a lab room, False for regular classrooms
   
   Constraints:
   - Cannot be used for two different courses in the same time slot
   - Lab courses must be scheduled in rooms with is_lab=True
   - Room capacity must be >= student group size
   
   Usage: Manhattan distance = |x1-x2| + |y1-y2| + |z1-z2|

================================================================================

5. StudentGroup
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Represents a group of students
   
   Attributes:
   - name (str): Name/code of the student group (e.g., "CSE-2027")
   - size (int): Number of students in the group
   - super_groups (list): List of parent/super groups this group belongs to
                         (e.g., CSE-2027 belongs to "Class of 2027")
   
   Hierarchy Example:
   - Class of 2027 (super group)
     ├── CSE-2027 (student group)
     ├── ME-2027 (student group)
     └── NT-2027 (student group)
   
   Constraints:
   - A student group cannot attend two courses in the same time slot
   - A super group cannot have both parent and child groups in same time slot
   - Only one lecture per day for each (course, student_group) pair

================================================================================

6. PreferenceBins (Data Structure)
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Maps time slots to preference bins
   
   Structure: Dictionary {slot_number: bin_id}
   
   Example:
   {
       1: 1, 2: 1, 3: 1,        # Slots 1-3 are in bin 1 (morning)
       4: 2, 5: 2, 6: 2,        # Slots 4-6 are in bin 2 (noon)
       7: 3, 8: 3, 9: 3         # Slots 7-9 are in bin 3 (evening)
   }
   
   Usage: Each CourseInstance has a preference_bin (1, 2, or 3).
          Penalties applied when actual_slot_bin differs from course_instance.preference_bin

================================================================================

7. ObjectiveFunctionWeights (Data Structure)
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Weights for the three penalty terms in fitness function
   
   Structure: List [w1, w2, w3]
   
   Components:
   - w1: Weight for Course Time Preference (CTP) penalty [formerly PTP]
         (each CourseInstance's preference_bin, not instructor preference)
   - w2: Weight for Room-to-Room Distance (RTR) penalty
   - w3: Weight for Course Stability penalty
   
   Usage: fitness = w1*ctp_penalty + w2*rtr_penalty + w3*stability_penalty

================================================================================

8. TimeTable (Data Structure)
   ────────────────────────────────────────────────────────────────────────────
   Purpose: Represents a complete weekly schedule (one chromosome in GA)
   
   Structure:
   {
       "Monday": {
           1: [course1, course2],      # Slot 1 has two courses
           2: [course3],               # Slot 2 has one course
           3: []                       # Slot 3 is empty
       },
       "Tuesday": { ... },
       ...
       "Sunday": { ... }
   }
   
   Components:
   - Key (string): Day of week
   - Value (dict): Maps slot_number (int) to list of courses
   
   Note: Multiple courses can be in the same slot if constraints allow
         (different instructors, rooms, student groups)

================================================================================
                           OBJECTIVE FUNCTIONS
================================================================================

1. Course Time Preference (CTP) Penalty (formerly PTP)
   ────────────────────────────────────────────────────────────────────────────
   Goal: Minimize deviation from each COURSE INSTANCE's preferred time bins
   
   Key Difference:
   - EACH CourseInstance has its OWN preference_bin (1=morning, 2=noon, 3=evening)
   - NOT inherited from the instructor's personal preference
   - Example: Prof Kumar's personal preference=1, but CS201-Lecture instance 
             they teach has preference_bin=3 (evening)
   
   Calculation:
   For each course instance in the timetable:
       penalty += |bin_of_assigned_slot - course_instance.preference_bin|
   
   Objective: Each course instance is scheduled according to its own preferred time bin
              (e.g., some lectures prefer morning, labs prefer afternoon, etc.)

================================================================================

2. Room-to-Room Distance (RTR) Penalty
   ────────────────────────────────────────────────────────────────────────────
   Goal: Minimize travel distance for students between consecutive classes
   
   Calculation:
   For each day, for each pair of consecutive time slots:
       For each student group appearing in both slots:
           distance = |x1-x2| + |y1-y2| + |z1-z2| (Manhattan distance)
           penalty += distance
   
   Also considers super groups:
   If student group X is in slot 1 and super group Y is in slot 2,
   the distance is calculated between their rooms.
   
   Objective: Reduce student travel time between back-to-back classes

================================================================================

3. Course Stability Penalty
   ────────────────────────────────────────────────────────────────────────────
   Goal: Keep same session type of a course in same slot and room throughout week
   
   Calculation:
   For each course and session type pair:
       If course appears multiple times across the week:
           For each occurrence after the first:
               if slot != first_slot: penalty += 1
               if room != first_room: penalty += 1
   
   Objective: All MA101 lectures should be at the same time and room every day

================================================================================
                              CONSTRAINTS
================================================================================

HARD CONSTRAINTS (Must be satisfied - violations rejected):

1. No Professor Conflict
   - Same instructor cannot teach two courses in same time slot

2. No Student Group Conflict
   - A student group cannot attend two courses in same slot
   - A super group cannot have both child and parent groups in same slot

3. No Room Conflict
   - A room cannot host two courses in same time slot

4. Lab Room Constraint
   - Lab sessions must be scheduled in rooms with is_lab=True

5. Room Capacity Constraint
   - Room capacity must be >= student group size

6. Slots Requirement Constraint
   - Each course must be scheduled for exactly slots_req number of times

7. Continuous Slots Constraint
   - If slots_continuous=True, all slots for a course on same day must be consecutive

8. Lecture Per Day Constraint
   - Each (course_id, student_group) can have at most one lecture per day
   - EXCEPTION: If lecture_consecutive=True, allow multiple lectures on same day
   - Use lecture_consecutive=True for courses requiring consecutive lectures (e.g., seminars, intensive courses)
   - Default is lecture_consecutive=False (max 1 lecture per day enforced)

================================================================================
                           SOFT CONSTRAINTS
================================================================================

These are incorporated into the objective function and weighted:

1. Course Time Preference (CTP)
   - Each CourseInstance has its own preferred time bin (1=morning, 2=noon, 3=evening)
   - Prefer assigned slot bin matches course instance's preference_bin
   - NOT based on instructor preference, but independent per course instance

2. Room-to-Room Travel (RTR)
   - Minimize student travel distance between consecutive classes

3. Course Session Consistency (Stability)
   - Keep same course sessions in consistent slot and room

================================================================================
                            DATA FLOW SUMMARY
================================================================================

Input:
├─ List of Courses (with instructor, room, student group, requirements)
├─ List of Rooms (with location, capacity, type)
├─ List of Student Groups (with hierarchy)
├─ Time Slots (number per day)
├─ Preference Bins (slot to bin mapping)
└─ Objective Function Weights

Process:
├─ GASearch.create_population()
│  └─ Generate valid timetables using smart greedy initialization
├─ GASearch.fitness()
│  ├─ _ptp_objective_function() [actually CTP - Course Time Preference]
│  ├─ _rtr_objective_function() [Room-to-Room Travel]
│  └─ _course_stability_objective_function() [Stability/Consistency]
├─ GASearch.select_parents()
├─ GASearch.crossover() / GASearch.mutate()
└─ Repeat for N generations

Output:
└─ Optimized TimeTable (Dictionary structure as described above)

================================================================================
                         RECENT SYSTEM UPDATES
================================================================================

As of April 2026, the system now supports:

1. Lecture Consecutiveness Control
   - lecture_consecutive parameter (boolean) on CourseInstance
   - Default: False (enforces max 1 lecture per day)
   - Set to True for intensive courses requiring multiple lectures in same day
   - Example: Design Thinking (DT101) with lecture_consecutive=True can have
             multiple lecture slots on same day

2. Elective Courses with Mixed Student Groups
   - Support for courses with student groups from multiple departments
   - Example: RL201 (Reinforcement Learning) for 35 students from CSE-2025 and ECE-2025
   - Elective groups can have multiple super_groups

3. Batch Courses for Large Student Bodies
   - Support for large auditorium-based courses
   - Example: DT101 (Design Thinking) for entire Class of 2025 (200 students)

4. Simplified Instructor Model
   - Removed personal preference from Instructor class
   - All time preferences now exclusively at CourseInstance level via preference_bin
   - Cleaner separation of concerns: Instructors are identifiers, Courses have preferences

================================================================================
                            EXAMPLE USAGE
================================================================================

# Initialize GA Search
ga = GASearch(
    time_slots=9,
    courses=[course1, course2, course3, ...],
    preference_bins={1:1, 2:1, 3:1, 4:2, 5:2, 6:2, 7:3, 8:3, 9:3},
    objective_function_weights=[1.0, 0.5, 0.8],
    rooms=[room1, room2, room3, ...],
    population_size=20,
    generations=100,
    mutation_rate=0.1
)

# Run genetic algorithm
population = ga.create_population()
# ... GA iterations ...
best_timetable = ga.run()

# best_timetable is a dictionary:
# {
#     "Monday": {1: [courses...], 2: [courses...], ...},
#     "Tuesday": {...},
#     ...
# }
